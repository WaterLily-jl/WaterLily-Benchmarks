# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

A benchmarking and profiling harness for [WaterLily.jl](https://github.com/WaterLily-jl/WaterLily.jl). It runs `sim_step!` across a matrix of (WaterLily version × Julia version × backend × thread count × case × case size × float type), saves results as JSON via `BenchmarkTools`, and compares them via `compare.jl`. It does **not** contain WaterLily itself — it consumes a checkout pointed to by `WATERLILY_DIR`.

## Prerequisites and environment

- `WATERLILY_DIR` must point to a local WaterLily.jl checkout. The shell scripts will `git checkout` inside that directory when `-w` is passed, so a clean working tree there is assumed. Pass `-wd <path>` to override.
- `Project.toml` declares `WaterLily` via a `[sources]` `path =` entry (currently hardcoded to `/home/b-fg/Workspace/tudelft1/WaterLily.jl`). The `update_environment` step in `benchmark.sh` re-`Pkg.develop`s `$WATERLILY_DIR` before each run, so this path entry is effectively overridden at runtime.
- Multi-Julia-version sweeps require `juliaup`; `benchmark.sh` calls `julia +$version` for non-default versions.
- GPU backends are auto-detected in `util.jl` by probing `nvcc`/`nvidia-smi` (CUDA) and `hipcc`/`rocm-smi` (ROCm). `using CUDA` / `using AMDGPU` is conditional on those checks — do not unconditionally import them.
- `LocalPreferences.toml` is regenerated by `benchmark.sh` per run: `Array` + `threads=1` → `backend = "SIMD"`, everything else → `backend = "KernelAbstractions"`. It is gitignored and should not be hand-edited for benchmark runs.
- On Ubuntu, `sh benchmark.sh` invokes `dash` and fails on the bash-isms in the script — invoke as `./benchmark.sh` or `bash benchmark.sh`. On Windows, use Git Bash. (Documented in README.)

## Common commands

```sh
# Quick default sweep (TGV + Jelly, current $WATERLILY_DIR, current Julia, Array + CuArray):
./benchmark.sh
julia --project compare.jl                          # prints tables to stdout

# Custom sweep — all case-aligned arrays (-c, -p, -s, -ft) must be the same length:
./benchmark.sh -v "release 1.11" -t "1 4" -b "Array CuArray" \
               -c "tgv jelly" -p "6,7 5,6" -s "25 25" -ft "Float32 Float64"

# Compare with speedup baseline and plots:
julia --project compare.jl --data_dir="data/benchmark" --plot_dir="plots" \
      --patterns="tgv jelly" --speedup_base="CPUx01" --sort=11

# Compare specific JSON files (mutually exclusive with --data_dir):
julia --project compare.jl benchmark_1.json benchmark_2.json --sort=7

# Profiling (NVIDIA only, requires WaterLily 'profiling' branch — profile.sh auto-checks it out):
./profile.sh -ns "nsys" -c "tgv sphere cylinder" -p "8 5 6" -s 1000 -r 1
./profile.sh -ns "ncu"  -c "cylinder" -p "4,5,6" -s 20 -k "327 355"
```

`-r/--run` for `profile.sh`: `0` = post-process only, `1` = run only, `2` = run then post-process.

## Architecture

Four small Julia files plus two bash drivers — flow is *bash sweeps args → spawns one `julia benchmark.jl` per cell of the matrix → that process writes one JSON*.

- `benchmark.sh` — outer driver. Parses CLA, loops `version × wl_version × backend × thread`, regenerates `LocalPreferences.toml`, calls `Pkg.develop` on `$WATERLILY_DIR`, then runs `julia benchmark.jl ...`. Each iteration is an independent Julia process; there is no shared state between cells beyond the JSON output.
- `benchmark.jl` — thin entry point. Calls `parse_cla` (from `util.jl`), calls `run_benchmarks`, which calls `add_to_suite!` (from `util.jl`) for each case, runs the suite with `samples=5, evals=1, gcsample=true`, and writes to `data/<hostname>_<git_hash>/<casename>_<sizes>_<maxsteps>_<ftype>_<backend>_<waterlilyHEADhash>_<juliaversion>.json`.
- `util.jl` — most of the actual logic lives here:
  - CLA parsing (`iarg`, `arg_value`, `metaparse`, `parsestringlist`, `parse_cla`).
  - `@add_benchmark` macro — wraps the kernel call with `KernelAbstractions.synchronize(backend)` so GPU timing is correct (without this you measure launch latency, not work).
  - `add_to_suite!` — for each grid size in `p`, instantiates the simulation, runs **50 warmup steps** (`sim_step!(sim, typemax(ft); max_steps=50)`), then registers the benchmarkable.
  - Backend detection (`_cuda`, `_rocm`) and the `backend_str` dict that produces table labels like `CPUx04`, `GPU-NVIDIA`, `GPU-AMD`. Thread count is read from `Threads.nthreads()` — set via `JULIA_NUM_THREADS` (the shell driver does this) or `julia -t N`.
  - `git_hash` is computed via `git -C $WATERLILY_DIR rev-parse --short HEAD` at load time; `find_git_ref(hash)` resolves it back to a tag/branch name if one matches. JSON filenames and result tags use the short hash.
  - Plotting helpers (`fancylogscale!`, `annotated_groupedbar`, `get_tickslogscale`) used by `compare.jl` and `profile.jl`.
  - `all_cases` and `tests_dets` (the domain-aspect-ratio dict used to compute DOF counts for cost-per-DOF) — keep these in sync when adding a case.
- `cases.jl` — each case is a function `(p, backend; T) -> Simulation` named exactly as the case string (`tgv`, `sphere`, `cylinder`, `donut`, `jelly`). They're looked up via `getf(case) = eval(Symbol(case))`, so the function name **must equal** the CLI case name. The `remeasure=true` flag is set for moving-body cases (`cylinder`, `jelly`) in `benchmark.jl` — match that pattern for any new moving-body case.
- `compare.jl` — loads benchmark JSONs (either via `--data_dir`+`--patterns` glob walk or as positional args, but **not both**), builds a `PrettyTable` per case per `log2p`, and optionally emits PDF cost and speedup plots. Speedup baseline is row 1 unless `--speedup_base="<backend>[,<wl_ref>][,<julia_version>]"` picks another. `--sort=N` sorts by table column index (1=backend … 11=speedup).
- `profile.jl` / `profile.sh` — parallel driver for NVTX-traced profiling. Requires the WaterLily `profiling` branch (which adds `NVTX.@range` around kernels). `profile.sh` force-checks-out that branch in `$WATERLILY_DIR` regardless of `-w`.

## Measurement methodology (what the table columns mean)

Default: `max_steps=25` per sample, `samples=5`, `evals=1`, `gcsample=true`, plus 50 warmup `sim_step!` calls before the benchmarkable is registered.

- **Min [s]** is the reported point estimate (robust to OS jitter/GC). Use this for speedup and cost-per-DOF, not Med.
- **Med/Max** expose spread. Samples reuse the same `Simulation`, so transient cases (sphere wake, jelly gait) naturally drift — treat per-row Δ within ±5% as noise unless Min/Max is also tight.
- **GC [%]** is the GC fraction of the *minimum-time* sample, not an average.
- **Cost [ns/DOF/dt]** = `tmin / N / max_steps` where `N = prod(tests_dets[case]["size"]) * 2^(3*p)`. If a new case has a non-cubic domain, fill in `tests_dets[case]["size"]` to make this meaningful.

## Adding a new case

1. Add the function in `cases.jl` returning a `Simulation`. Name it exactly as the CLI string you want to use.
2. Add the string to `all_cases` and add an entry under `tests_dets` in `util.jl` (`size` = domain aspect tuple, `title` = plot title).
3. If the case has a moving body, add it to the `remeasure=...` whitelist in `benchmark.jl`.

## Output layout and reproducibility notes

- Each Julia invocation writes one JSON into `data/<hostname>_<wl_short_hash>/`. `hostname` is `gethostname()` and `wl_short_hash` is the WaterLily HEAD when `util.jl` is loaded — so checking out a different WaterLily ref between sweeps produces a separate subdirectory.
- The JSON filename encodes all axes; `compare.jl` reads structure from the embedded `BenchmarkGroup` tags, not the filename. Renaming files is safe; editing tags is not.
- `data/`, `plots/`, `Manifest.toml`, and `LocalPreferences.toml` are gitignored.
